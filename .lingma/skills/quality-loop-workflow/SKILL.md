# Quality-Loop 多智能体协作工作流

## 概述

这是一个基于**四层质量保障**的自动化协作工作流，通过循环调用四个专业智能体来完成复杂任务的高质量交付：

1. **Planning-Agent**（规划智能体）：深度研读源码，防止重复造轮子，拆分问题并制定验收标准
2. **Implementation-Agent**（编码智能体）：严格按照规划完整实现，绝不使用TODO或简化
3. **QA-Agent**（验收智能体）：根据规划文档对代码进行功能验收，智能判断回退路径
4. **CodeReview-Agent**（代码审查智能体）：深度审查代码质量，谨慎维护项目约定文件

### 核心特性

- **约定驱动**：通过 `user-conventions.md` 和 `project-conventions.md` 确保编码不脱离项目思想
- **智能循环**：验收失败时根据问题类型智能选择回退路径（Implementation或Planning）
- **动态子规划**：任何Agent发现模块复杂度过高时都可触发子工作流
- **质量保障**：四层检查（规划→实现→验收→审查）确保高质量交付

## 工作流程

### ⚠️ 核心约束：单次循环粒度控制

**重要原则**：每一次完整的工作流循环（阶段1→2→3→4）只处理**一个模块**。

**为什么需要这个约束**：
- 避免单次规划内容过多导致上下文溢出
- 确保每个模块都能得到充分的关注和细致的实现
- 提高验收和审查的准确性和针对性
- 便于问题定位和返工修正

**如何判断"一个模块"**：
- 通常对应一个 `plan_module_Y.md` 文件
- 包含明确的功能边界和代码范围
- **代码量指导原则**：
  - L1 简单模块：< 300 行（单文件或小功能）
  - L2 中等模块：300-800 行（多文件但逻辑清晰）
  - L3 复杂模块：800-1500 行（需要进一步拆分为子模块）
  - > 1500 行：强烈建议拆分为多个子模块
- **灵活性说明**：代码量只是参考，关键看功能是否独立、逻辑是否清晰。如果模块功能单一且内聚，即使超过1000行也可以作为一个模块处理

### 阶段 0：初始化与约定文件检查

在开始之前，首先检查约定文件是否存在：

**检查清单**：
- [ ] `.lingma/conventions/user-conventions.md` 是否存在？
- [ ] `.lingma/conventions/project-conventions.md` 是否存在？

**如果不存在**：
- Planning-Agent 在首次规划时会提示创建
- 或者手动创建基础版本（见"约定文件初始化"章节）

**简单任务**（单一文件修改、小功能添加）：
- 直接使用 Implementation-Agent 完成即可
- 无需启动完整工作流

**复杂任务**（多模块开发、系统级改造）：
- 必须启动完整的四智能体循环工作流
- 按照以下步骤执行

---

### 阶段 1：架构规划（Planning-Agent）

**触发条件**：
- 接收到新的复杂任务（首次调用）
- QA-Agent 报告需要重新规划
- Implementation-Agent 请求子规划

**执行步骤**：

1. **确定本次循环的处理范围**
   - 如果是首次调用：对整个任务进行全局分析和模块拆解
   - 如果是后续调用：仅针对当前需要处理的**单个模块**进行详细规划

2. **调用 Planning-Agent 智能体**
   ```
   指令示例（首次调用）：
   "请使用 planning-agent 对 [任务描述] 进行全局架构分析和模块拆解"
   
   指令示例（单模块规划）：
   "请使用 planning-agent 对模块 [Y] 进行详细规划。这是当前循环的唯一任务，请提供该模块的完整实现细节、依赖关系和验收标准。"
   ```

3. **等待产出物**：
   - `.plans/phase_X/plan_module_Y.md`（模块规划文档）
   - `.plans/phase_X/acceptance_criteria_Y.md`（验收标准文档）
   - `.plans/phase_X/progress_module_Y.json`（进度跟踪文件，初始状态为PENDING）

4. **验证规划完整性**：
   - 检查规划文档是否足够详细（应包含具体的函数签名、类结构、算法逻辑等）
   - 确认该模块的依赖关系已明确定义
   - 验证验收标准是否可量化、可检查
   - 确认规划中指出了可复用的现有资源

5. **如果规划不完整**：
   - 要求 Planning-Agent 补充缺失部分
   - 重复此阶段直到规划完整

---

### 阶段 2：编码实现（Implementation-Agent）

**触发条件**：
- Planning-Agent 已完成**当前模块**的详细规划
- QA-Agent 已调度**下一个待处理的单个模块**

**执行步骤**：

1. **获取当前任务**：
   - 读取对应的 `plan_module_Y.md` 和 `acceptance_criteria_Y.md`
   - 读取 `.lingma/conventions/project-conventions.md`
   - **禁止**同时处理多个模块的实现

2. **调用 Implementation-Agent 智能体**
   ```
   指令示例：
   "请使用 implementation-agent 执行模块 [Y] 的实现。这是本次循环的唯一任务，请完整实现该模块的所有功能，不要留 TODO。规划文件路径：[.plans/phase_X/plan_module_Y.md]"
   ```

3. **等待产出物**：
   - 完整的源代码文件（无 TODO、无占位符）
   - 更新的 `progress_module_Y.json`（状态标记为 IMPLEMENTED）

4. **验证实现完整性**：
   - 检查代码文件是否已创建且内容完整
   - 确认进度文件已更新为 IMPLEMENTED 状态
   - 快速扫描是否有明显的 TODO 或占位符注释
   - **如果发现问题**：立即要求 Implementation-Agent 修正，不要进入验收阶段

5. **如果实现不完整**：
   - 要求 Implementation-Agent 修正
   - 重复此阶段直到实现完整

---

### 阶段 3：质量验收（QA-Agent）

**触发条件**：
- Implementation-Agent 报告**当前模块**完成

**执行步骤**：

1. **调用 QA-Agent 智能体**
   ```
   指令示例：
   "请使用 qa-agent 验收刚完成的模块 [Y]。这是本次循环的唯一验收任务。验收通过后，请明确指出下一步是进入CodeReview阶段还是需要回退。"
   ```

2. **等待验收结果**：

   **情况 A：验收通过**
   - 检查 `progress_module_Y.json` 状态是否为 VERIFIED
   - QA-Agent 应返回：
     - ✅ 验收通过的确认信息
     - 📋 下一步：进入 CodeReview 阶段（阶段4）
   
   **情况 B：验收失败 - 必须回退修正**
   - ⚠️ **重要**：验收失败时，**绝对不能**进入 CodeReview 阶段
   - 记录具体问题清单（精确到文件和行号）
   - 根据问题类型决定回退路径：
     - **命名/规范类** 或 **功能缺失类** → **回退到 Implementation-Agent**（阶段2）
     - **架构/设计类** 或 **规划缺陷类** → **回退到 Planning-Agent**（阶段1）
   - **修正流程**：
     1. 回退到对应的 Agent 进行修正
     2. 修正完成后，重新从阶段2（Implementation）或阶段1（Planning）开始
     3. 再次进入阶段3（QA）进行验收
     4. 重复此循环直到验收通过
   - **重复验收直到通过**：不要跳过或妥协验收标准
   - **连续3次失败** → 上报人工介入

3. **循环控制**：

   | 循环次数 | 处理方式 |
   |---------|---------|
   | 第1次失败 | 正常回退修正 |
   | 第2次失败 | QA-Agent 必须详细分析问题根源 |
   | 第3次失败 | 强制上报人工介入 |

---

### 阶段 4：代码审查（CodeReview-Agent）

**触发条件**：
- ⚠️ **仅在 QA-Agent 验收通过后才可进入此阶段**
- QA-Agent 返回的状态必须是 `VERIFIED`
- 如果 QA 验收失败，必须先回退修正，重新验收通过后才能进入

**执行步骤**：

1. **调用 CodeReview-Agent 智能体**
   ```
   指令示例：
   "请使用 codereview-agent 对已通过验收的模块 [Y] 进行深度代码审查。检查代码质量、废弃代码，并谨慎评估是否需要更新 project-conventions.md。"
   ```

2. **等待审查结果**：
   - 质量评分（可读性、可维护性、性能、安全性、一致性）
   - 发现的问题清单（按严重程度分类）
   - 废弃代码识别
   - 项目约定更新（如有）

3. **验证审查完整性**：
   - 确认 CodeReview-Agent 已检查所有维度
   - 如果更新了 `project-conventions.md`，确认更新理由充分
   - 确认提供了具体的改进建议

4. **输出最终报告**：
   ```markdown
   ## 模块 [Y] 完成报告
   
   ### 质量评估
   - 总体评分：X.X/5
   - 发现问题：X个（X个严重，X个警告，X个建议）
   
   ### 约定更新
   - ✅ 已更新 / ❌ 未更新 project-conventions.md
   
   ### 建议
   - [列出关键建议]
   
   ### 状态
   - ✅ 模块 [Y] 已完成并通过审查
   ```

## 智能循环策略

### 完整工作流程图

```
【阶段1】Planning-Agent 规划
    ↓
【阶段2】Implementation-Agent 实现
    ↓
【阶段3】QA-Agent 验收
    ├─ ✅ 通过 → 【阶段4】CodeReview-Agent 审查 → 完成
    └─ ❌ 失败 → 问题分类
                  ├─ 命名/规范类 → 回退到【阶段2】Implementation 修正
                  ├─ 功能缺失类 → 回退到【阶段2】Implementation 修正
                  ├─ 架构/设计类 → 回退到【阶段1】Planning 重新规划
                  └─ 规划缺陷类 → 回退到【阶段1】Planning 重新规划
                      ↓
                  修正后重新进入【阶段2】→ 【阶段3】QA 再次验收
                      ↓
                  重复循环直到 QA 验收通过（最多3次）
```

**关键原则**：
- ✅ **只有 QA 验收通过才能进入 CodeReview**
- ❌ **QA 验收失败绝对不能跳过修正直接进入 CodeReview**
- 🔄 **验收失败必须回退到 Implementation 或 Planning 进行修正**

### QA验收失败时的决策树

```
QA发现问题
    ↓
问题分类
    ├─ 命名/规范类 → 回退到 Implementation-Agent
    │                  └─ 修正后重新进入 QA（阶段3）
    │
    ├─ 功能缺失类 → 回退到 Implementation-Agent
    │                  └─ 补充实现后重新进入 QA（阶段3）
    │
    ├─ 架构/设计类 → 回退到 Planning-Agent
    │                  └─ 重新规划后进入 Implementation（阶段2）
    │
    └─ 规划缺陷类 → 回退到 Planning-Agent
                       └─ 修正规划后进入 Implementation（阶段2）

循环终止条件：
- 验收通过 → 进入 CodeReview（阶段4）
- 连续3次失败 → 上报人工介入
```

### 问题类型判断指南

| 问题类型 | 特征 | 回退目标 | 示例 |
|---------|------|---------|------|
| 命名/规范类 | 代码写对了但风格不对 | Implementation | 变量命名不符合规范 |
| 功能缺失类 | 漏写了某些功能 | Implementation | 缺少异常处理分支 |
| 架构/设计类 | 实现方式与规划偏离 | Planning | 使用了错误的算法 |
| 规划缺陷类 | 规划本身有问题 | Planning | 未考虑边界条件 |

**判断原则**：
- 如果是**实现层面**的问题（代码写错了、漏写了）→ Implementation
- 如果是**设计层面**的问题（规划错了、需求变了）→ Planning

## 子规划工作流触发

### 触发条件（任何Agent都可触发）

| 触发Agent | 触发场景 | 处理方式 |
|----------|---------|---------|
| Planning | 任务复杂度过高（L3级） | 将当前模块拆分为子模块，为每个子模块生成独立规划 |
| Implementation | 实现过程中发现子功能过于复杂 | 暂停当前实现，请求Planning生成子规划 |
| QA | 验收时发现某个功能点需要细化 | 标记该功能点，请求Planning生成细化规划 |
| CodeReview | 审查时发现架构需要重构 | 建议重新规划，由用户决定是否触发 |

### 子工作流执行流程

```
主工作流 [模块Y]
    ↓
发现需要子规划（例如：子功能Y.1过于复杂）
    ↓
【子工作流启动】
    ├─ Planning-Agent 生成子规划
    │   └─ 输出：plan_module_Y_1.md + acceptance_criteria_Y_1.md
    ├─ Implementation-Agent 实现子功能Y.1
    ├─ QA-Agent 验收子功能Y.1
    └─ CodeReview-Agent 审查子功能Y.1
    ↓
【子工作流完成】
    ↓
回到主工作流，继续处理模块Y的其他部分
```

### 嵌套限制

- 最多允许**2层嵌套**（主工作流 → 子工作流 → 孙工作流）
- 超过2层必须上报人工介入
- Planning-Agent 在生成子规划时必须评估是否真的需要

## 约定文件管理

### 约定文件初始化

如果项目中还没有约定文件，在首次使用工作流时需要创建：

**1. 创建 user-conventions.md**（由用户手动编写）

```markdown
# 用户约定

> 本文档由项目成员手动维护，表达项目的核心价值观和编码偏好。

## 1. 项目思想

### 1.1 设计哲学
- [描述项目的核心设计理念]

### 1.2 技术选型原则
- [说明技术选型的标准和偏好]

## 2. 编码约定

### 2.1 命名规范
- [列出命名规则]

### 2.2 代码风格
- [说明代码风格要求]

## 3. 测试要求
- [说明测试覆盖率和类型要求]

## 4. 文档要求
- [说明文档编写标准]
```

**2. 创建 project-conventions.md**（初始为空，由CodeReview-Agent逐步填充）

```markdown
# 项目约定

> 本文档由CodeReview-Agent自动维护，记录经过验证的最佳实践和规范。
> 最后更新时间：[日期]
> 版本：v1.0

## 1. 架构规范
[由CodeReview-Agent自动填充]

## 2. 代码质量规范
[由CodeReview-Agent自动填充]

## 3. 命名约定
[由CodeReview-Agent自动填充]

## 历史变更记录

### v1.0 ([日期])
- 初始版本
```

### CodeReview-Agent 更新约定的原则

**三次反思**：
1. **必要性**：这条规范是否真的需要写入约定？
2. **通用性**：这条规范是否具有普适性？
3. **精简性**：约定文件是否因此变得臃肿？

**更新条件**：
- ✅ 应该写入：多次出现的共性问题、影响团队协作的规范
- ❌ 不应写入：单次偶发问题、过于具体的业务逻辑

**文件大小控制**：
- 目标：`project-conventions.md` 保持在 **500行以内**
- 如果超限，必须合并或删除过时内容

### 示例 2：处理验收失败

```
【阶段 3：质量验收】
正在调用 qa-agent 验收模块 2...
❌ 验收失败：发现 3 处命名不规范

【问题分类】
- 问题类型：命名/规范类
- 回退目标：Implementation-Agent

【返回阶段 2：修正实现】
正在调用 implementation-agent 修正模块 2...
[修正后重新提交]

【再次验收】
```

## 关键注意事项

### 必须遵守的规则

1. **严格按顺序执行**：
   - 必须先规划，再编码，然后验收，最后审查
   - 禁止跳过任何一个阶段
   - 每次循环只处理一个模块

2. **保持上下文连贯**：
   - 每个Agent都要读取前一个阶段的产出
   - Implementation-Agent 必须查阅规划和约定
   - QA-Agent 必须对照验收标准检查
   - CodeReview-Agent 必须参考项目约定

3. **循环直到合格**：
   - 验收失败必须返工，不能妥协
   - 只有完全符合规范才能进入下一阶段
   - 连续3次失败必须人工介入

4. **约定文件谨慎更新**：
   - CodeReview-Agent 必须进行三次反思
   - 每次更新必须说明理由
   - 保持文件精简（500行以内）

### 常见陷阱与避免方法

| 陷阱 | 症状 | 解决方法 |
|------|------|----------|
| 规划不充分 | 编码时发现大量未定义的需求 | 返回阶段1补充规划 |
| 验收松懈 | 代码中有明显问题但通过了 | 严格执行验收标准 |
| **验收失败后跳过修正** | **QA失败后直接进入CodeReview** | **必须回退到Implementation或Planning修正，重新验收** |
| 循环失控 | 同一模块反复失败超过3次 | 暂停工作流，人工介入 |
| 约定污染 | project-conventions.md变得臃肿 | 定期审计和精简 |
| 上下文溢出 | 一次规划太多模块 | 严格遵守单模块循环原则 |

## 故障排查

### 问题 1：工作流卡住不动

**可能原因**：
- 某个Agent没有正确响应
- 进度文件状态不一致

**解决方法**：
1. 检查最新的 `progress_module_*.json` 文件
2. 确认所有文件路径正确
3. 手动调用相应的Agent重试

### 问题 2：验收一直失败

**可能原因**：
- 规划文档本身有问题
- 验收标准过于严苛或不清晰

**解决方法**：
1. 返回阶段1，让 Planning-Agent 重新审查规划
2. 检查验收标准是否可量化、可检查
3. 如果连续失败3次，暂停并人工介入

### 问题 3：代码实现偏离规划

**可能原因**：
- Implementation-Agent 没有仔细阅读规划文档
- 规划文档不够清晰

**解决方法**：
1. QA-Agent 明确指出偏离的具体位置
2. 要求 Implementation-Agent 重新阅读规划
3. 必要时让 Planning-Agent 优化规划文档的表述

### 问题 4：约定文件变得臃肿

**可能原因**：
- CodeReview-Agent 过于频繁地更新约定
- 没有进行精简性检查

**解决方法**：
1. 立即停止更新，进行约定文件审计
2. 合并相似的规范
3. 删除过时的内容
4. 用更简洁的语言重述规范

### 问题 5：QA验收失败后错误地进入CodeReview

**症状**：
- QA-Agent 报告验收失败
- 但工作流直接调用了 CodeReview-Agent
- 代码中的问题没有被修正

**可能原因**：
- 没有正确理解验收失败后的回退流程
- 误以为 CodeReview 可以修复验收发现的问题

**解决方法**：
1. **立即停止**当前的 CodeReview 调用
2. 查看 QA-Agent 的验收报告，确认问题类型
3. 根据问题类型回退：
   - 命名/功能问题 → 调用 Implementation-Agent 修正
   - 架构/规划问题 → 调用 Planning-Agent 重新规划
4. 修正完成后，**重新从阶段2或阶段1开始**
5. 再次进入阶段3（QA）进行验收
6. **只有验收通过后**才能进入阶段4（CodeReview）

**记住**：CodeReview-Agent 只审查已通过验收的代码，不负责修复验收发现的问题。

## 成功指标

### 质量指标

- **代码一致性**：95%以上的代码符合 project-conventions.md
- **重复实现率**：低于5%（通过Planning阶段的源码研读保证）
- **验收通过率**：首次验收通过率≥70%，二次验收通过率≥95%
- **约定文件质量**：保持在500行以内，每条规范都有实际案例支撑

### 效率指标

- **平均循环次数**：每个模块平均1.5次循环内完成
- **子规划触发率**：10%-20%的模块需要子规划
- **人工介入率**：低于5%的任务需要人工介入

### 可维护性指标

- **约定文件更新频率**：每周不超过2次
- **CodeReview发现问题数**：平均每模块3-5个问题
- **废弃代码识别率**：每次Review至少检查一次废弃代码

## 总结

Quality-Loop 工作流的核心价值在于：

1. **分工明确**：四个Agent各司其职，发挥最大效能
2. **质量保障**：通过四层检查确保每个环节都符合规范
3. **可追溯性**：所有进度和决策都有文档记录
4. **智能循环**：根据问题类型精准回退，提高效率
5. **约定驱动**：通过双层约定文件确保编码不脱离项目思想
6. **动态适应**：支持子规划触发，灵活应对复杂任务

记住：**质量优先于速度**。宁可多花时间在规划、验收和审查上，也不要产出不合格的代码。**约定优先于便利**：严格遵守项目约定，确保代码的一致性和可维护性。
