# Quality-Loop 工作流快速上手指南

## 5分钟快速开始

### 第一步：检查约定文件

在使用工作流之前，确保项目中有两个约定文件：

```bash
# 检查文件是否存在
ls .lingma/conventions/user-conventions.md
ls .lingma/conventions/project-conventions.md
```

**如果不存在**，请创建基础版本：

**user-conventions.md**（由你手动编写）：
```markdown
# 用户约定

## 1. 项目思想
- [填写你的项目设计理念]

## 2. 编码约定
- [填写你的编码规范]
```

**project-conventions.md**（初始为空）：
```markdown
# 项目约定

> 本文档由CodeReview-Agent自动维护

## 1. 架构规范
## 2. 代码质量规范
## 3. 命名约定

## 历史变更记录
### v1.0 (初始版本)
```

---

### 第二步：启动工作流

当你有一个复杂的开发任务时，使用以下命令：

```
请使用 quality-loop-workflow 处理 [你的任务描述]
```

**示例**：
```
请使用 quality-loop-workflow 为Quick Actions添加一个插件配置管理系统
```

---

### 第三步：观察执行过程

工作流会自动按以下顺序执行：

```
【阶段 1】Planning-Agent 规划
    ↓
【阶段 2】Implementation-Agent 实现
    ↓
【阶段 3】QA-Agent 验收
    ↓
【阶段 4】CodeReview-Agent 审查
```

每个阶段完成后，你会看到详细的报告。

## 常见场景

### 场景 1：简单任务（不需要工作流）

**适用情况**：
- 修改单个文件
- 添加一个小功能
- 修复一个简单的bug

**做法**：
直接使用 Implementation-Agent，无需启动完整工作流。

```
请使用 implementation-agent 修改 SearchBar.tsx，添加搜索历史记录功能
```

---

### 场景 2：复杂任务（需要完整工作流）

**适用情况**：
- 开发一个新模块
- 重构多个文件
- 系统级改造

**做法**：
启动完整的 Quality-Loop 工作流。

```
请使用 quality-loop-workflow 重构整个搜索系统，支持拼音搜索和模糊匹配
```

---

### 场景 3：验收失败需要修正

**发生情况**：
QA-Agent 发现代码不符合要求。

**自动处理**：
工作流会根据问题类型自动回退：
- 命名/功能问题 → 回退到 Implementation-Agent
- 架构/规划问题 → 回退到 Planning-Agent

**你只需要**：
观察验收报告，等待自动修正完成。

---

### 场景 4：需要子规划

**发生情况**：
某个模块过于复杂，需要进一步拆分。

**自动处理**：
任何Agent都可以触发子规划工作流，自动处理子模块。

**你只需要**：
观察子工作流的执行进度。

## 关键概念

### 四个Agent的职责

| Agent | 职责 | 产出 |
|-------|------|------|
| **Planning-Agent** | 研读源码、制定规划 | 规划文档 + 验收标准 |
| **Implementation-Agent** | 按照规划完整实现 | 源代码文件 |
| **QA-Agent** | 对照标准验收代码 | 验收报告 |
| **CodeReview-Agent** | 深度审查代码质量 | 审查报告 + 更新约定 |

### 两个约定文件

| 文件 | 维护者 | 内容 |
|------|--------|------|
| **user-conventions.md** | 你（手动） | 项目思想、编码偏好 |
| **project-conventions.md** | CodeReview-Agent | 经过验证的最佳实践 |

### 智能循环策略

```
验收失败
    ↓
问题分类
    ├─ 实现问题 → 回退到 Implementation
    └─ 设计问题 → 回退到 Planning
```

## 最佳实践

### ✅ 推荐做法

1. **在 user-conventions.md 中明确项目思想**
   - 这会让所有Agent更好地理解你的需求
   - 例如："本项目以规则驱动为核心，避免硬编码"

2. **定期查看 project-conventions.md 的更新**
   - 了解CodeReview-Agent发现了哪些最佳实践
   - 确认新增的规范是否合理

3. **对于复杂任务，耐心等待完整流程**
   - 不要中途打断工作流
   - 让每个Agent充分发挥作用

4. **关注验收和审查报告**
   - 这些报告揭示了代码的质量问题
   - 可以帮助你改进后续的规划

### ❌ 避免做法

1. **不要在规划不充分时就急于编码**
   - Planning-Agent 的源码研读很重要
   - 可以避免重复造轮子

2. **不要让验收失败超过3次**
   - 如果连续失败，说明有根本性问题
   - 需要人工介入分析

3. **不要忽视 CodeReview-Agent 的建议**
   - 即使是"建议"级别的问题也值得关注
   - 可以提升代码质量

4. **不要手动编辑 project-conventions.md**
   - 这个文件由 CodeReview-Agent 自动维护
   - 手动编辑可能导致冲突

## 故障排除

### Q1: 工作流卡住了怎么办？

**检查**：
1. 查看最新的 `progress_module_*.json` 文件
2. 确认是否有Agent正在等待输入

**解决**：
- 如果是等待输入，提供所需信息
- 如果是卡死，手动调用相应的Agent重试

---

### Q2: 验收一直失败怎么办？

**检查**：
1. 查看验收报告中的具体问题
2. 分析问题类型（实现问题还是设计问题）

**解决**：
- 第1-2次失败：让工作流自动回退修正
- 第3次失败：人工介入，检查规划是否有问题

---

### Q3: 如何知道工作流是否完成？

**标志**：
- 所有模块的状态都是 `REVIEWED`
- 看到最终完成报告
- 没有待处理的任务

**查看进度**：
```bash
# 查看所有模块的进度
cat .plans/phase_*/progress_*.json
```

---

### Q4: 约定文件变得太大怎么办？

**检查**：
```bash
# 查看行数
wc -l .lingma/conventions/project-conventions.md
```

**解决**：
- 如果超过500行，联系 CodeReview-Agent 进行精简
- 合并相似的规范
- 删除过时的内容

## 进阶技巧

### 技巧 1：自定义验收标准

在 Planning-Agent 生成验收标准后，你可以手动补充：

```markdown
# acceptance_criteria_Y.md

## 额外要求
- [ ] 必须支持国际化
- [ ] 响应时间 < 100ms
```

---

### 技巧 2：强制触发子规划

如果你知道某个模块很复杂，可以提前要求：

```
请使用 planning-agent 对模块 [Y] 进行规划，并评估是否需要子规划
```

---

### 技巧 3：审查特定维度

你可以要求 CodeReview-Agent 重点关注某个方面：

```
请使用 codereview-agent 审查模块 [Y]，特别关注性能和安全问题
```
