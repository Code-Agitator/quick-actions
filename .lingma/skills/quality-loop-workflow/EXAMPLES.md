## 示例 4：CodeReview-Agent 更新项目约定

### 场景描述

在审查多个模块后，CodeReview-Agent 发现了一个共性模式，决定更新项目约定。

### 执行过程

#### 审查模块 A

```
【模块 A 审查】
codereview-agent 输出：
✅ 审查完成

发现的问题：

1. 第45行：Promise 未处理 rejection

💡 建议：
2. 建议使用统一的错误处理模式
```

#### 审查模块 B

```
【模块 B 审查】
codereview-agent 输出：
✅ 审查完成

发现的问题：
🟡 警告：
1. 第78行：async 函数未包含 try-catch
2. 第120行：Promise chain 缺少 .catch()
```

#### 审查模块 C

```
【模块 C 审查】
codereview-agent 输出：
✅ 审查完成

发现的问题：
🔴 严重问题：
1. 第34行：未处理的 Promise rejection 导致静默失败
```

#### 三次反思与约定更新

```
【CodeReview-Agent 内部决策】

识别潜在规范点：异步函数必须包含错误处理

第一次反思：必要性
- ✅ 应该写入：在3个模块中都发现了类似问题，是共性问题
- 影响：未处理的 Promise rejection 会导致静默失败，难以调试

第二次反思：通用性
- ✅ 应该写入：适用于所有异步代码，不限于特定模块
- 普适性：这是 JavaScript/TypeScript 的通用最佳实践

第三次反思：精简性
- ✅ 可以接受：可以用简洁的语言表述
- 当前 project-conventions.md 有320行，新增后约325行，仍在500行以内

决定：更新 project-conventions.md
```

#### 更新约定文件

```
【更新 project-conventions.md】

新增内容：

## 2. 代码质量规范

### 2.1 错误处理
- 所有异步函数必须包含try-catch或.catch()
- 禁止空的catch块，必须记录日志或重新抛出
- API调用失败必须返回统一的Error对象

【版本历史更新】

## 历史变更记录

### v1.2 (2026-05-09)
- 新增：异步函数错误处理规范
- 更新理由：在模块A、B、C的审查中发现多处未处理的Promise rejection，可能导致静默失败
```

#### 审查报告

```
【模块 C 审查报告】

### 项目约定更新
- ✅ 已更新 `project-conventions.md`
  - 新增规范：「异步函数必须包含错误处理」
  - 更新位置：第2章 代码质量规范 > 2.1 错误处理
  - 更新理由：本次review及前两次review共发现5处未处理的Promise rejection

### 总结
```

## 总结

通过这些示例，你可以看到：

1. **正常流程**：规划 → 实现 → 验收 → 审查，一气呵成
2. **验收失败**：智能回退到合适的阶段修正
3. **规划缺陷**：回退到 Planning-Agent 重新规划
4. **约定更新**：CodeReview-Agent 谨慎地维护项目规范
5. **子规划**：灵活应对复杂模块
6. **简单任务**：不需要启动完整工作流
7. **异常情况**：连续失败时人工介入

Quality-Loop 工作流的核心理念是：**质量优先于速度，约定优先于便利**。通过四层检查和智能循环，确保每次交付都是高质量的。
